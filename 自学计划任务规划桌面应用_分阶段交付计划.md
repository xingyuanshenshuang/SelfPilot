# 自学计划任务规划桌面应用 — 分阶段交付计划

> 基于 PRD v1.0（2026-06-15）制定  
> 制定日期：2026-06-23

---

## 一、优先级划分

| 优先级 | 定义 | 包含模块 |
| :--- | :--- | :--- |
| **P0** | 核心可用：用户能创建目标、自动拆解、完成/跳过任务、查看今日待办与进度 | 目标总览、目标树（创建/拆解/完成/跳过）、今日待办、本地数据存储 |
| **P1** | 体验增强：提升管理效率与可视化 | 日历视图、数据统计（基础图表）、鼓励语库、设置（主题/导出导入） |
| **P2** | 锦上添花：高级功能与预测 | 数据统计（预测完成时间）、日历热力图、鼓励语个性化规则（连续天数高亮） |

## 二、分阶段 Sprint 交付计划

| 迭代阶段 | 功能模块 | 核心操作路径（用户如何操作） | 验收标准（Given-When-Then 格式） |
| :--- | :--- | :--- | :--- |
| **Sprint 1** | 创建目标与自动拆解 | 用户在目标树页面点击【+创建目标】，输入名称、截止日期、总量及单位，点击【自动拆解计划】。 | **Given** 用户已输入有效目标信息（名称、未来截止日期、总量>0），**When** 点击【自动拆解计划】，**Then** 系统按"剩余天数（不含今天，且≥1）"将总量平均分配到每天生成任务（余数分配到前几天），每个任务记录计划数量和单位，并在目标树下按日期平铺展示。 |
| **Sprint 1** | 今日待办与任务完成 | 用户在目标总览页面查看"今日待办"列表，勾选布尔型任务或点击数量型任务的【完成】按钮输入实际完成量。 | **Given** 今日有待办任务，**When** 用户完成一个布尔型任务（勾选）或数量型任务（输入实际完成量），**Then** 该任务状态更新为✅（100%）或🟡（部分完成），目标总览中的对应目标进度百分比实时重新计算，且今日待办列表状态同步更新。 |
| **Sprint 1** | 任务跳过 | 用户在今日待办或目标树中点击任务旁的【跳过】按钮。 | **Given** 任务处于未完成状态，**When** 用户点击【跳过】，**Then** 任务标记为"已跳过"，不计入完成量，且**不影响**后续任何自动调整或进度计算。 |
| **Sprint 2** | 补完成（历史任务） | 用户在目标总览的"逾期任务"区或目标树的历史日期中找到已跳过/逾期的任务，点击【补完成】并输入实际完成量。 | **Given** 用户选择了一个历史日期的已跳过或逾期任务，**When** 点击【补完成】并输入实际完成量，**Then** 系统更新该历史任务的完成记录和累计进度，但**不会重新计算**未来未完成任务的计划数量（未来任务保持原拆解数量不变）。 |
| **Sprint 2** | 手动触发重新规划 | 用户在目标详情页点击【重新规划剩余时间】按钮。 | **Given** 目标下存在未完成的任务（不含已跳过的），**When** 用户手动点击【重新规划剩余时间】，**Then** 系统将目标下所有未完成任务（不含跳过）按新的剩余天数重新平均分配计划数量，并提示用户确认变更后的任务列表。 |
| **Sprint 2** | 阶段（二级节点）管理 | 用户在目标树下右键点击目标，选择【添加阶段】，输入阶段名称；可将现有任务拖拽或编辑归属到阶段下。 | **Given** 用户已创建一个目标，**When** 添加阶段并将任务归入该阶段，**Then** 目标树显示"目标 → 阶段 → 任务"层级，阶段进度按子任务完成百分比汇总，支持阶段的编辑与删除（删除时提示子任务转为独立任务或级联删除）。 |
| **Sprint 3** | 日历视图（月/周/日） | 用户点击左侧导航【日历视图】，查看月网格中每日的任务分布，点击某日展开当天任务列表。 | **Given** 用户已进入日历视图，**When** 切换月/周/日视图或点击某日期，**Then** 正确展示该日期下所有任务（自动拆解+周期性手动任务）及其状态（✅/🟡/🔲/⚠️逾期），支持批量标记完成/跳过。 |
| **Sprint 3** | 数据统计（基础图表） | 用户点击左侧导航【数据统计】，查看各目标完成百分比柱状图和近7天/30天趋势折线图。 | **Given** 系统已有任务完成记录，**When** 用户进入数据统计页面，**Then** 正确渲染各目标完成百分比柱状图，以及近7天/30天完成任务数量趋势折线图（数量型按完成数量统计，布尔型计1）。 |
| **Sprint 4** | 鼓励语库与基础规则 | 用户点击左侧导航【鼓励语库】，查看预设列表并添加自定义鼓励语；完成当日首个任务后弹出鼓励语。 | **Given** 鼓励语库中已有预设或自定义语句，**When** 用户当日首次完成任务，**Then** 弹窗或顶部提示一句随机鼓励语；连续完成天数按"每天至少完成一个任务"标准正确累计。 |
| **Sprint 4** | 设置与数据管理 | 用户点击左侧导航【设置】，切换浅色/深色主题，或导出/导入本地数据（JSON/SQLite）。 | **Given** 用户在设置页面，**When** 切换主题或执行导出/导入操作，**Then** 主题实时生效；导出生成本地备份文件；导入时若ID冲突弹出对话框让用户选择覆盖/跳过/重命名。 |
| **Sprint 5** | 日历热力图 | 用户在数据统计页面查看日历热力图。 | **Given** 系统已有历史任务数据，**When** 用户查看热力图，**Then** 按日期格子颜色深浅正确展示每日"完成任务量 / 当日应有任务总量"的比例。 |
| **Sprint 5** | 鼓励语个性化规则 | 用户连续打卡达到3天或7天时，收到进阶或高亮鼓励语；完成全部目标时收到庆祝语。 | **Given** 用户连续完成天数达到阈值，**When** 完成任务触发鼓励语时，**Then** 优先从对应等级语库中抽取（1天普通、3天进阶、7天高亮、全部完成庆祝）。 |
| **Sprint 6** | 预测完成时间 | 用户在数据统计页面查看基于过去7天平均速度的完成预测。 | **Given** 目标有足够历史完成数据，**When** 用户查看预测模块，**Then** 系统基于过去7天平均完成速度，计算并提示每个目标是否能按期完成，给出"需提高速度"或"可提前完成"等建议。 |

## 三、重点逻辑差异分析与开发注意事项

### 3.1 补完成 vs. 重新规划 —— 核心差异

| 维度 | 补完成（Backfill） | 手动触发重新规划（Re-plan） |
| :--- | :--- | :--- |
| **触发时机** | 用户主动对历史日期的已跳过/逾期任务进行补录 | 用户主动点击目标详情页的【重新规划剩余时间】按钮 |
| **作用对象** | 仅针对单个历史任务的完成记录 | 针对目标下**所有未完成任务**的计划数量 |
| **对未来任务的影响** | **绝不重算**：未来未完成任务的计划数量保持原拆解值不变 | **强制重算**：按新的剩余天数重新平均分配所有未完成任务的数量 |
| **对进度的影响** | 更新累计完成量，从而更新目标总进度百分比 | 更新未来任务的计划数量，可能改变目标剩余工作量 |
| **开发坑点** | 容易误写成"补完成后自动调整后续任务"。必须明确：**补完成只写历史完成记录，不触发任何重新分配算法。** | 重新规划时必须排除"已跳过"的任务；重新分配后需保留用户曾手动修改过的单日数量（PRD未明确，建议弹出确认框展示变更前后对比）。 |

### 3.2 开发需要特别注意的坑

| 坑点 | 说明 | 建议处理 |
| :--- | :--- | :--- |
| **补完成不联动未来** | 最容易出现的 Bug 是补完成后系统自动把剩余总量重新分摊到未来天数。这与 PRD 明确要求的"补完成不会重新计算未完成任务的计划数量"直接冲突。 | 在补完成逻辑中严格隔离：只更新 `actual_completed` 字段和进度汇总，禁止调用任何 `redistribute_remaining()` 方法。 |
| **重新规划时跳过任务的处理** | 重新规划必须过滤掉"已跳过"的任务，否则会把用户明确放弃的任务又分配进计划。 | 重新规划查询条件中增加 `status != 'skipped'`，并在 UI 提示用户"已跳过的任务不参与重新规划"。 |
| **重新规划后手动修改的丢失** | 用户可能之前手动调整过某些日期的计划数量，重新规划会覆盖这些手动值。 | 重新规划前展示变更预览（Before/After），让用户确认；或在技术设计上将"手动修改"标记为 `is_manual=true`，重新规划时保留手动项。 |
| **部分完成的向上汇总精度** | 数量型任务的完成百分比 = 实际完成量 / 计划数量，需精确到父级目标和阶段的加权平均。 | 阶段/目标的进度计算应基于子任务"计划数量 × 完成百分比"加权，而非简单算术平均。 |
| **连续完成天数的边界** | PRD 规定"每天至少完成一个任务"才计入连续；若当天无任何任务，则"不中断也不计入"。 | 判断逻辑应为：`if (当天有任务 && 至少完成1个) 连续+1; else if (当天无任务) 连续不变; else 连续归零`。避免将"无任务日"误判为中断。 |

---

*本计划按 P0 → P1 → P2 顺序分 6 个 Sprint 交付，每个 Sprint 建议周期为 1~2 周。*
